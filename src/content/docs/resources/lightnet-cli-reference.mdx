`@lightnet/cli` provides command-line tools for working with LightNet projects.

## Install

From the root of your LightNet project, add the CLI as a development dependency:

```bash
pnpm add -D @lightnet/cli@latest
```

## Usage

Run LightNet CLI commands with:

```bash
pnpm lightnet <command> [options]
```

Show all available commands:

```bash
pnpm lightnet --help
```

Show help for a specific command:

```bash
pnpm lightnet <command> --help
```

## R2 setup

R2 commands and R2 file checks require `rclone`. You can [download it from rclone.org](https://rclone.org/install/).

The first time you run a command that needs R2 access, LightNet prompts for any missing bucket settings and saves them in `lightnet-cli.json`.
You can also create or edit that file manually.

LightNet reads R2 connection settings from `lightnet-cli.json`:

```json
{
  "r2": {
    "publicUrl": "https://files.your-site.org",
    "accountId": "your-cloudflare-account-id",
    "accessKeyId": "your-r2-access-key-id",
    "bucket": "your-bucket-name"
  }
}
```

Add `lightnet-cli.json` to your git repository so every developer and CI job uses the same non-secret bucket settings.

These settings match the fields from [Configure file storage](/deploy/file-storage/#configuration-fields):

- `publicUrl`: the public URL for your files, such as `https://files.your-site.org`
- `accountId`: your Cloudflare account ID
- `accessKeyId`: the public access key ID from your Cloudflare token
- `bucket`: the R2 bucket for your site

LightNet passes these settings to `rclone` automatically. You do not need to create an `rclone` remote manually.

The secret access key is intentionally not saved in `lightnet-cli.json`.

For local interactive use, you can let the CLI prompt for the secret access key. The prompt stores the key only for the current command session.

For CI, scripts, or repeated local commands, set the secret access key in your shell before running the command:

```bash
export LIGHTNET_R2_SECRET_ACCESS_KEY="your-r2-secret-access-key"
```

<Aside type="caution">
Commit `lightnet-cli.json`, but keep `LIGHTNET_R2_SECRET_ACCESS_KEY` in your shell, CI secrets, or password manager. 
Do not commit the secret access key to your repository.
</Aside>

## Available commands

### `check-translations`

```bash
pnpm lightnet check-translations
```

Use this command to find translations that are still missing your site's languages.

Run it from the root of your LightNet site project.

The command asks whether to run `pnpm build` for you. When you confirm, the build runs as part of the check and the translation results are read from that build.

Use it when you want to:

- catch incomplete translations before deploying
- review translation work after adding a new language
- check inline translation maps after editing content or templates

#### Common checks

Check translations:

```bash
pnpm lightnet check-translations
```

Run a build before checking translations:

```bash
pnpm lightnet check-translations --build
```

Check translations without running a build or showing a prompt:

```bash
pnpm lightnet check-translations --no-build
```

#### Options

- `--build`: runs `pnpm build` before checking translations.
- `--no-build`: skips the build prompt and checks the latest available build output.

If the command reports missing translations, add the missing values to the content or template that owns them.

<Aside>
This command only validates translations used during the last build.
Missing translations are not reported for labels that were never referenced while building the site.
</Aside>

### `check-files`

```bash
pnpm lightnet check-files
```

Use this command to find broken file links and unused files in a LightNet site.

Run it from the root of your LightNet site project.

#### Common checks

Check all local content files and thumbnails:

```bash
pnpm lightnet check-files
```

Check only thumbnails:

```bash
pnpm lightnet check-files --scope thumbnails
```

Check only downloadable content files:

```bash
pnpm lightnet check-files --scope content-files
```

Check content files in Cloudflare R2:

```bash
pnpm lightnet check-files --r2
```

Remove unused files after reviewing the prompt:

```bash
pnpm lightnet check-files --fix
```

Remove unused files without an interactive confirmation prompt:

```bash
pnpm lightnet check-files --fix-without-confirm
```

Check only content files in Cloudflare R2:

```bash
pnpm lightnet check-files --r2 --scope content-files
```

#### Options

- `--fix`: removes unused files or R2 objects. It does not repair broken links.
- `--fix-without-confirm`: removes unused files or R2 objects. Skips the deletion confirmation prompt ⚠️.
- `--r2`: checks content files in Cloudflare R2 instead of local files under `public/files`. Thumbnail checks still use local files.
- `--scope <values>`: limits the check to `content-files`, `thumbnails`, or both, separated by commas. By default, both scopes are checked.

#### What to fix

The command reports two kinds of issues:

- **Missing referenced content files**: a content item links to a file that is not available. Fix these by uploading or restoring the missing file, or by updating the content item to point at the right file.
- **Orphaned files or thumbnails**: a file exists, but no content item or category uses it. Review these before deleting them, especially in shared buckets.

For local checks, content file links should point to files under `public/files` and use `/files/...` URLs.

For thumbnail checks, use `./images/...` references next to your media or category content. Media thumbnails live under `src/content/media/images`, and category thumbnails live under `src/content/categories/images`.
If your site does not have categories, you do not need to create that directory just for this check.

When a missing content file is reported, the output includes the content JSON filename that references it. Use that filename as your starting point for fixing the link.

#### Reading the result

If everything is clean, the command prints nothing and exits successfully.

If it finds broken links or unused files, it prints grouped sections with counts in the headings and exits with an error status.
That makes it safe to use in CI: a failing check means there is something to review before deployment.

The sections can include:

1. warnings
2. missing referenced content files
3. orphaned local content files or orphaned R2 objects
4. orphaned media thumbnails
5. orphaned category thumbnails
6. removed items

#### Checking R2

Use `--r2` when your content files are stored in Cloudflare R2 instead of `public/files`.

R2 checks require an R2 connection configured through rclone. See [R2 setup](#r2-setup) for details.

If a saved R2 connection stops working, rerun the command and follow the prompt to refresh the saved settings.

R2 checks review every object in the bucket. Files such as `.DS_Store` can be reported as unused objects if they are present.

#### Removing unused files

Use `--fix` when you want the CLI to remove unused local files or unused R2 objects for you.

Fix mode does not repair broken links. If a content item points at a missing file, update that content item or restore the file yourself.

Before deleting anything, the CLI shows the files or objects that will be removed and asks for confirmation. Use `--fix-without-confirm` only when you are comfortable skipping that prompt, such as in a trusted CI cleanup job.

If all reported issues were unused files and they were removed successfully, the command exits successfully. If broken links still remain, the command continues to exit with an error so you can fix them before deployment.

### `check-links`

```bash
pnpm lightnet check-links
```

Use this command to check whether the URLs used in your media items are still reachable online.

Run it from the root of your LightNet site project.

The command reads every URL from the `content` array in `src/content/media/**/*.json`, including uploaded file links and external links. For each unique URL, it sends a lightweight request to check whether the target is available without downloading large files such as videos or PDFs.

Use it when you want to:

- find broken media links before visitors hit them
- catch deleted or moved external pages
- verify hosted PDFs, audio files, and video files without downloading the full file
- keep large LightNet content collections healthy

#### Relationship to `check-files`

Checking links for files managed by the LightNet site overlaps with what `check-files` does, but the commands answer different questions.

Use `check-files` to audit your site's file storage. It checks whether local files under `public/files` or files in Cloudflare R2 match the file URLs used by your content, and it can also report unused files.

Use `check-links` to audit what visitors can reach online. It checks each content URL, including external pages, external media, and LightNet-managed files after they have a public URL.

Run both when you want full coverage: `check-files` keeps your managed file collection consistent, and `check-links` catches public availability problems such as deleted external pages, moved files, host errors, or a deployed file URL that no longer responds.

#### Common checks

Check all media links:

```bash
pnpm lightnet check-links
```

Adjust the per-link timeout:

```bash
pnpm lightnet check-links --timeout 15000
```

#### Options

- `--timeout <milliseconds>`: sets the per-link timeout. The default is `10000` milliseconds, or 10 seconds. Use a higher value for slow hosts or large collections that include links to slower external services.

#### Site URL

Root-relative URLs need a site URL before they can be checked online:

```json
{
  "type": "upload",
  "url": "/files/my-pdf.pdf"
}
```

The first time `check-links` finds a root-relative URL, the CLI asks for your site URL and saves it in `lightnet-cli.json`:

```json
{
  "siteUrl": "https://example.org"
}
```

After that, `/files/my-pdf.pdf` is checked as `https://example.org/files/my-pdf.pdf`.

#### Reading the result

If all links are available, the command exits successfully.

If a link is broken, the command prints the URL, the error or HTTP status, and the media JSON file that references it. 
Fix the URL in that content file, or restore the missing target, and then rerun the check.

Some websites block automated checks even though the URL works in a browser. These are reported as protected or forbidden warnings instead of hard failures.

### `r2`

```bash
pnpm lightnet r2 <command>
```

Use this command to list, copy, move, and remove files in a Cloudflare R2 bucket.
Run it anywhere inside your LightNet site project.

This command requires an R2 connection configured through rclone. See [R2 setup](#r2-setup) for details.

Use it when you want to:

- [list](#list-files) the files in your site's R2 bucket
- [copy](#copy-files) local files or folders to R2
- [download](#copy-files) files from R2
- [move](#move-files) files or folders inside R2
- [remove](#remove-files) individual files or folders from R2

#### List files

```bash
pnpm lightnet r2 ls [path]
```

Lists files in the R2 bucket. The optional path limits the listing to a folder.

List all files:

```bash
pnpm lightnet r2 ls
```

List files under a folder:

```bash
pnpm lightnet r2 ls images
```

Pipe the plain output into another command:

```bash
pnpm lightnet r2 ls images | grep jpg
```

#### Copy files

```bash
pnpm lightnet r2 cp [options] <source> <destination>
```

Copies files between your local filesystem and R2, or from one R2 path to another.

Use the `r2:` prefix for R2 paths. At least one side must be an R2 path.

Upload a local folder to R2:

```bash
pnpm lightnet r2 cp ./images r2:images
```

Upload the contents of the current directory:

```bash
pnpm lightnet r2 cp . r2:
```

Download an R2 folder locally:

```bash
pnpm lightnet r2 cp r2:images ./images
```

Download from R2 into the current directory:

```bash
pnpm lightnet r2 cp r2:images .
```

Copy a folder inside R2:

```bash
pnpm lightnet r2 cp r2:images r2:archive/images
```

The command uses `rclone copy`. When copying a directory, it copies the contents of that directory into the destination.

When the destination file or directory already exists, the command follows Unix-style copy overwrite behavior and prompts before replacing existing files. 
Use `-f` or `--force` to skip overwrite prompts.

Skip overwrite prompts:

```bash
pnpm lightnet r2 cp -f ./images r2:images
```

#### Options

- `-f`, `--force`: skips overwrite confirmation prompts.

#### Move files

```bash
pnpm lightnet r2 mv [options] <source> <destination>
```

Moves or renames files and folders inside the configured R2 bucket.

Both paths are R2 paths in the configured bucket. The `r2:` prefix is accepted but optional. Local paths like `./image.jpg` are rejected.

Rename a file:

```bash
pnpm lightnet r2 mv image.jpg image-backup.jpg
```

Move a folder:

```bash
pnpm lightnet r2 mv images archived-images
```

The command follows Unix-style move behavior. It can move files or directories without `-r` or `--recursive`.

When the destination file or directory already exists, the command prompts before replacing it. 
Prefix overwrites replace the destination prefix contents before moving. Bucket-root moves are refused.

Skip overwrite prompts:

```bash
pnpm lightnet r2 mv -f images archived-images
```

#### Options

- `-f`, `--force`: skips overwrite confirmation prompts.

#### Remove files

```bash
pnpm lightnet r2 rm <path>
```

Deletes a single R2 file. Confirmation is required by default.

Delete one file:

```bash
pnpm lightnet r2 rm images/photo.jpg
```

Skip confirmation:

```bash
pnpm lightnet r2 rm -f images/photo.jpg
```

#### Remove directories

Use `-r` or `--recursive` to remove a directory:

```bash
pnpm lightnet r2 rm -r images/old
```

Skip confirmation for recursive removal:

```bash
pnpm lightnet r2 rm -rf images/old
```

Remove all content from the bucket:

```bash
pnpm lightnet r2 rm -r /
```

Be careful with recursive deletion. It removes matching objects under that R2 path.

When removing `/`, the command deletes all content in the configured R2 bucket. 
This always prompts for confirmation, even when you pass `-f` or `--force`.

#### Options

- `-f`, `--force`: skips the deletion confirmation prompt, except when removing `/`.
- `-r`, `--recursive`: removes all matching objects under a directory.
