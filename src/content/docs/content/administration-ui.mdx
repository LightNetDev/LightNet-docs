LightNet stores site content in JSON files. That works well for developers, but it can be less convenient for non-technical editors.
To make editing easier, LightNet includes an Administration User Interface (Administration UI).
The Administration UI provides a graphical editor for creating, updating, and deleting media items and media collections.
Under the hood, it uses [Sveltia CMS](https://github.com/sveltia/sveltia-cms).

Changes made in the Administration UI are written either to your local file system or to a Git host.

<Aside title="Experimental" type="caution">
The Administration UI is still experimental.
It is safe to use in real projects, but it still lacks some quality-of-life improvements, such as more comprehensive validation for invalid content changes.
</Aside>

## Add the Administration UI to your project

Follow these steps to add the Administration UI to your LightNet project:

<Aside>
Your template may already include the Administration UI. For example, `example-template` ships with it preinstalled.
You can still follow this guide from the beginning if you want to verify the setup.
</Aside>

<Steps>
1. **Install the integration**: From your project folder, install the LightNet Administration UI integration (`@lightnet/sveltia-admin`):
    ```bash
    pnpm add @lightnet/sveltia-admin
    ```
    Then add the integration to `astro.config.mjs`.
1. **Add the minimum configuration**: Update `astro.config.mjs` to include both the LightNet integration and the Administration UI integration. Here is a minimal example:

    ```js title="astro.config.mjs" ins={1,10}
    import lightnetSveltiaAdmin from "@lightnet/sveltia-admin"
    import { defineConfig } from "astro/config"
    import lightnet from "lightnet"

    export default defineConfig({ 
      integrations: [
        lightnet({
            //...
        }),
        lightnetSveltiaAdmin({})
      ]    
    });
    ```
</Steps>

## Edit languages through `languages.json`

The Administration UI can also edit LightNet languages through a `languages.json` file at the root of your site.
This is controlled by the experimental `useLanguagesCollection` option in `lightnetSveltiaAdmin(...)`.

Enable the experimental option in `astro.config.mjs`:

```ts title="astro.config.mjs"
lightnetSveltiaAdmin({
  experimental: {
    useLanguagesCollection: true,
  },
})
```

When enabled, the Administration UI expects a `languages.json` file at the root of your LightNet site.

<Aside type="caution" title="Experimental option">
This workflow is optional and experimental.
The option name and behavior may still change before the feature is finalized.
</Aside>

### Extract `languages.json`

If your site currently defines `languages` inline inside `lightnet(...)`, move that array into a dedicated `languages.json` file and then import it back into `astro.config.mjs`.

Before:

```ts title="astro.config.mjs"
import { defineConfig } from "astro/config"
import lightnet from "lightnet"

export default defineConfig({
  integrations: [
    lightnet({
      languages: [
        {
          code: "en",
          label: {
            en: "English",
            de: "Englisch"
          },
          isDefaultSiteLanguage: true,
        },
        {
          code: "de",
          label: {
            en: "German",
            de: "Deutsch",
          },
          isSiteLanguage: true,
        },
      ],
    }),
  ],
})
```

After:

```ts title="astro.config.mjs" {1,8}
import languages from "./languages.json"
import { defineConfig } from "astro/config"
import lightnet from "lightnet"

export default defineConfig({
  integrations: [
    lightnet({
      languages
    }),
  ],
})
```

Create `languages.json` at the site root with the same array:

```json title="/languages.json"
[
  {
    "code": "en",
    "label": {
      "en": "English",
      "de": "Englisch"
    },
    "isDefaultSiteLanguage": true
  },
  {
    "code": "de",
    "label": {
      "en": "German",
      "de": "Deutsch"
    },
    "isSiteLanguage": true
  }
]
```

## Run the Administration UI

The Administration UI can save changes either to your local file system or to a Git host such as GitHub.

### Accessing your local file system

Use this workflow when developing locally on your own computer.

<Steps>
1. **Start the development server**: Run this command from your project folder:
    ```bash
    pnpm dev
    ```
    This starts the LightNet development server.
2. **Open the Administration UI**: In Chrome or another Chromium-based browser, go to [localhost:4321/admin](http://localhost:4321/admin).
3. **Choose the local workflow**: Click the "Work with Local Repository" button.
4. **Grant folder access**: When your browser prompts you, select your project folder and allow access.
</Steps>

<Aside type="note">
The local admin workflow requires the File System Access API, which is currently available only in Chrome and other Chromium-based browsers.
Brave disables this API by default; you can enable it at `brave://flags/#file-system-access-api`.

The online admin workflow works in all major browsers.
</Aside>


### Accessing a Git host

The Administration UI also supports editing content online.
It does this by connecting to a Git host such as GitHub.
When used this way, content changes are written as Git commits through the Git host's web API.
Authentication is handled by the Git host as well, so you can manage editor access through repository permissions.

This architecture has several advantages:
- No need to deploy and maintain a custom backend API.
- Security is handled by the Git host, including support for two-factor authentication.
- Git history makes it easy to review changes and roll back when needed.
- Git repositories can often be hosted at low or no cost.

LightNet works with either GitHub and GitLab as Git hosts. We recommend GitHub for new LightNet projects.

<Aside>
GitHub does not yet support client-only PKCE authentication for this workflow.
For now, use an auth proxy such as [sveltia/sveltia-cms-auth](https://github.com/sveltia/sveltia-cms-auth).
GitHub is planning native support for this flow in the future. Track progress in [GitHub roadmap issue #1153](https://github.com/github/roadmap/issues/1153).
</Aside>

#### Connect to GitHub

Follow these steps to connect the Administration UI to a GitHub repository.

<Steps>
1. **Host your repository on GitHub**: Make sure your LightNet project is hosted on GitHub. If you still need to create or move the repository, follow the [hosting guide](/deploy/hosting).
1. **Deploy an auth proxy**: Set up [`sveltia/sveltia-cms-auth`](https://github.com/sveltia/sveltia-cms-auth) on Cloudflare Workers. Follow the instructions for setting up GitHub authentication in the [sveltia-cms-auth README](https://github.com/sveltia/sveltia-cms-auth/blob/main/README.md). Copy the public base URL of the deployed proxy.
1. **Open your project configuration**: Return to your project and open `astro.config.mjs`.
1. **Configure the backend**: Add the `backend` configuration for your GitHub repository:
    ```js title="astro.config.mjs" ins={6-10}
    // ...
    export default defineConfig({ 
      integrations: [
        // ...
        lightnetSveltiaAdmin({
            backend: {
                name: "github",
                repo: "<Owner>/<Repository Name>",
                baseUrl: "https://your-auth-proxy.workers.dev",  
            },
        })
      ]    
    });
    ```
    Set `repo` to the GitHub repository path and `baseUrl` to the public URL of your auth proxy service.
1. **Use the Administration UI**: After your site is deployed, open `/admin` on your website, click the login button, and authorize access with GitHub.
    You can now edit media items online. If other people also need access, add them as collaborators or grant access through your GitHub organization.
</Steps>

<Aside>
Even if you configure a GitHub backend, you can still use a local Administration UI workflow during development.
</Aside>

## Reference

You can pass the following options to the `lightnetSveltiaAdmin` integration inside `astro.config.mjs`:

```js title="astro.config.mjs" {9}
import lightnet from "lightnet";
import lightnetSveltiaAdmin from "@lightnet/sveltia-admin";
import { defineConfig } from "astro/config";

export default defineConfig({
  integrations: [
    lightnet({...}),
    lightnetSveltiaAdmin({
      /* add config here */
    }),
  ],
});
```

### `path`

type: `string` \
example: `"admin"` \
required: `false` \
default: `"admin"`

The path where the Administration UI is available on your site. Change this path to make your Git host connection less guessable.

<Aside type="caution">
To protect your site, change the default "admin" path to something less guessable.
</Aside>

### `maxFileSize`

type: `number` \
example: `15` \
required: `false` \
default: `25`

The maximum file size, in megabytes, that can be uploaded. The default is 25 MB, which aligns with Cloudflare's
maximum file size.
Lower this if you want editors to stay within a stricter content policy, or raise it only when your hosting and storage setup can support larger uploads.

### `backend`

type: `object` \
example: `{ name: "github", repo: "your-org/your-site", baseUrl: "https://your-auth-proxy.workers.dev" }` \
required: `false`

This option configures the Git host the Administration UI connects to. LightNet currently supports GitLab and GitHub backends.
Its fields map directly to the Sveltia CMS settings for GitHub or GitLab backends.
Use this when editors should make changes through your hosted repository instead of accessing the project folder locally.

#### `backend: "github"`

GitHub backend configuration has these options:

- **name**: `"github"` (required) selects GitHub as the backend. Use this when your LightNet site is stored on GitHub.
- **repo**: `string` (required) the GitHub repository path of your site. This tells the Administration UI exactly which repository to read from and commit to.
- **branch**: `string` the branch used by the Administration UI. The default is `"main"`. Set this when your production content lives on a different default branch or when you want editors to work against a dedicated content branch.
- **baseUrl**: `string` the URL of the service that handles the authentication request. This is useful because GitHub currently needs an auth proxy for this workflow.

See the [Sveltia CMS documentation](https://sveltiacms.app/en/docs/backends/github) for more details.

<Aside>
GitHub does not yet support client-only PKCE for this flow.
For now, use an auth proxy such as [`sveltia/sveltia-cms-auth`](https://github.com/sveltia/sveltia-cms-auth) and point `baseUrl` to that service.
</Aside>

#### `backend: "gitlab"`

GitLab backend configuration has these options:

- **name**: `"gitlab"` (required) selects GitLab as the backend.
- **repo**: `string` (required) the GitLab repository path of your site.
- **appId**: `string` the Application ID from the site's GitLab settings.
- **branch**: `string` the branch used by the Administration UI. The default is `"main"`.
- **authType**: `"pkce"` the authentication type to use. This is the default and currently the only supported option.

See the [Sveltia CMS documentation](https://sveltiacms.app/en/docs/backends/gitlab) for more details.


### `experimental`

type: `object` \
example: `{ useLanguagesCollection: true }` \
required: `false`

Optional experimental features for the Administration UI.
If omitted, all experimental features remain disabled.
Use this section when you want to try newer admin capabilities before they become part of the stable default setup.

#### `experimental.useLanguagesCollection`

type: `boolean` \
example: `true` \
required: `false` \
default: `false`

Enables Administration UI editing for LightNet languages through a root-level `languages.json` file.
When this option is enabled, keep your `lightnet({ languages })` configuration and import the array from `./languages.json` in `astro.config.mjs`.
The `languages.json` file must live at the root of the site repository.
This is useful when non-developers need to manage language labels or enabled locales through the Administration UI instead of editing `astro.config.mjs` directly.

#### `experimental.hideCategoriesCollection`

type: `boolean` \
example: `true` \
required: `false` \
default: `false`

Hides the `categories` collection from the Administration UI.
Use this when your taxonomy is managed only by developers, or when editors should assign existing categories without creating or editing category definitions.

#### `experimental.hideMediaTypesCollection`

type: `boolean` \
example: `true` \
required: `false` \
default: `false`

Hides the `media-types` collection from the Administration UI.
Use this when your media types are part of a controlled site structure and editors should only choose from predefined types instead of maintaining them.

#### `experimental.hideMediaCollectionsCollection`

type: `boolean` \
example: `true` \
required: `false` \
default: `false`

Hides the `media-collections` collection from the Administration UI.
Use this when collection structure is planned by the site team and ordinary editors only need to work on media items.

#### `experimental.hideAuthorsField`

type: `boolean` \
example: `true` \
required: `false` \
default: `false`

Hides the `authors` field in the media item editor.
Use this when author metadata is not relevant for your library.

#### `experimental.hideCategoriesField`

type: `boolean` \
example: `true` \
required: `false` \
default: `false`

Hides the `categories` field in the media item editor.
Use this when categories are not relevant for your library.

#### `experimental.showContentLabelField`

type: `boolean` \
example: `false` \
required: `false` \
default: `true`

Shows the `content[].label` field in the media item editor.
Keep this enabled when editors need a reader-friendly label for uploaded files or links instead of showing raw filenames or URLs. 
Disable it when you want the content editor to stay as minimal as possible.

#### `experimental.showDateCreatedField`

type: `boolean` \
example: `false` \
required: `false` \
default: `true`

Shows the `dateCreated` field in the media item editor.
Enable this when editors should control when an item appears to have been added to the library. 
Disable it when you want LightNet to handle the date automatically and avoid manual date mistakes.

#### `experimental.showCommonIdField`

type: `boolean` \
example: `false` \
required: `false` \
default: `true`

Shows the `commonId` field in the media item editor.
Enable this when editors manage translated versions of the same media item and need to group them together across languages. 
Disable it when your site is single-language or when translation grouping is handled by developers.

#### `experimental.showSlugField`

type: `boolean` \
example: `false` \
required: `false` \
default: `true`

Shows the `slug` field editor for new entries.
Enable this when editors need direct control over entry URLs and filenames. 
Disable it when you want slugs to be generated automatically from the entry title or name, 
which reduces the chance of inconsistent slugs.

#### `experimental.fileStorage`

type: `object` \
example: `{ name: "cloudflare-r2", accessKeyId: "your-access-key-id", ... }` \
required: `false`

External file storage configuration for Administration UI uploads.

Use this when your Git repository should not be used for file storage, or when your hosting setup needs larger uploads. For example, Cloudflare Workers only allows files up to 25 MB.

LightNet currently supports Cloudflare R2 for this option.
Because this feature is experimental, the configuration may change in future releases.
See [Setup large file storage](/deploy/file-storage) for a full walkthrough.

#### `experimental.fileStorage.name`

type: `"cloudflare-r2"` \
required: `true`

Selects Cloudflare R2 as the upload storage backend.

#### `experimental.fileStorage.accessKeyId`

type: `string` \
required: `true`

The R2 Access Key ID used for uploads.

#### `experimental.fileStorage.bucket`

type: `string` \
required: `true`

The R2 bucket name used to store uploaded files.

#### `experimental.fileStorage.accountId`

type: `string` \
required: `true`

Your Cloudflare account ID. LightNet uses it to construct the R2 API endpoint.

#### `experimental.fileStorage.publicUrl`

type: `string` \
required: `true`

Public base URL used for previews and downloads of uploaded files.
This is required because the R2 S3 API itself always requires authentication.

#### `experimental.fileStorage.prefix`

type: `string` \
example: `"uploads/"` \
required: `false`

Optional path prefix inside the bucket where uploaded files should be stored.
